## 遇到的问题和解决

##### 问题一：前后端文档与技术栈不一致

早期草稿中技术栈提及 Spring Boot，与实际确定的 Flask + MySQL 不符。  
**解决：** 统一以 `backend/` 目录下实际代码为准，在 `docs/architecture.md`、`docs/database.md` 和 `CLAUDE.md` 中全部更正为 Flask + MySQL（可切 SQLite），并与前端同学在 `docs/api.md` 中约定统一的接口路径与响应格式。

##### 问题二：打卡记录中 `plan_id` 的必要性

最初设计 `training_record` 时，曾考虑 `plan_id` 必填以保证"先建计划再打卡"的逻辑，但这样会导致用户临时起意的自由训练无法记录。  
**解决：** 将 `plan_id` 设为可空，保留 `user_id` 必填作为打卡记录的强制归属键；`plan_id` 为空时视为自由训练，通过 ER 图图例和文档说明解释这一设计意图，避免协作同学误解。

##### 问题三：MySQL 与 SQLite 开发环境切换

搭档本地没有装 MySQL，或在不同电脑上开发时数据库环境不一致。  
**解决：** 所有数据库操作统一通过 SQLAlchemy ORM，`backend/.env` 中只需改 `DATABASE_URL` 即可切换引擎，`backend/models/` 中的代码零改动。

## 心得体会

通过这次架构设计，深刻体会到"先理清数据关系再动手"的重要性。在划分 `users / training_plan / training_record / knowledge` 四张表时，先在 ER 图上确认每张表的主键和外键流向，比直接写 model 再返工节省了大量时间。尤其是 `training_record` 上同时保留 `user_id` 和可空 `plan_id` 的设计，如果没有提前在文档中说明业务含义，很容易被搭档误以为是冗余设计。

另一个收获是环境配置与代码分离：`DATABASE_URL` 通过 `.env` 注入而非硬编码，使开发、联调、生产三套环境可以共用同一套 ORM 代码，减少了"在我电脑上能跑"的问题。文档与 `.env` 示例保持同步，也方便自己和搭档各自独立启动项目，降低协作成本。

后续若接入真实大模型 API，只需在 `services/` 下扩展 `ai_service`，并在配置中补充 API key，无需推翻后端整体架构，分层设计的优势会随着项目复杂度增加愈发明显。
